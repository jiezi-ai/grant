# 治理规则

## G1. 政策文件范围

以下文件属于政策/章程，修改需走变更流程：
- `policy/*`（所有政策文件）
- `README.md`（项目主页）
- `BUDGET.md`（预算规划）
- `TIMELINE.md`（时间线）
- `stages/*`（阶段说明）
- `docs/*`（申请指南和模板）

以下文件属于记录，日常更新无需走变更流程：
- `records/*`（支出、批次、里程碑、变更记录）

学生申请数据存储在数据库中，不在仓库文件里。

## G2. 章程变更流程

1. 从 main 创建分支，命名 `charter/变更描述`
2. 修改相关政策/章程文件
3. 在 `records/changelog.md` 中添加记录，包含：
   - 编号（递增，格式 `CNNN`）
   - 日期
   - 变更内容
   - 原因
   - 影响范围（已参与的 / 未来的 / 所有人）
   - 生效时间（立即 / 下批起 / 具体日期）
4. 提交 PR，标题格式：`[章程] 变更描述`
5. Review 后合并

## G3. Checkpoint 反思机制

规则不是一成不变的。我们设置 checkpoint，当数据触发时进行公开反思和规则调整。

### 漏斗转化 checkpoint

| 触发条件 | 反思内容 |
|----------|----------|
| 第一阶段累计满 50 人 | 申请流程是否顺畅，PR 模板是否清晰 |
| 第二阶段申请者累计满 5 人 | 筛选门槛是否合理，50 star 要求是否恰当 |
| 某批申请数不足 50 | 宣传是否不够，项目吸引力是否下降 |
| 第一阶段 → 第二阶段转化率低于 5% | 门槛可能过高，需评估是否调整 |
| 第一阶段 → 第二阶段转化率高于 20% | 门槛可能过低，需评估是否调整 |

### 异常信号 checkpoint

| 触发条件 | 反思内容 |
|----------|----------|
| 某批 star 刷量率超过 20% | star 门槛设计是否在诱导作弊 |
| 大量申请者 edu 邮箱验证失败 | 身份验证方式是否需要调整 |
| 供应商价格变动超过 20% | 是否需要更换供应商或调整预算 |

### 定期 checkpoint

- **每批结束后**：发布公开复盘，回顾该批数据，评估是否需要调整规则
- 复盘内容记录在 `records/` 中

### checkpoint 触发后的流程

1. 发起人发布公开复盘，说明触发了哪个 checkpoint、数据是什么
2. 如需调整规则，按 G2 章程变更流程执行（创建分支 → 修改 → changelog → PR）
3. 所有调整对已参与者的影响需在 PR 中明确说明
